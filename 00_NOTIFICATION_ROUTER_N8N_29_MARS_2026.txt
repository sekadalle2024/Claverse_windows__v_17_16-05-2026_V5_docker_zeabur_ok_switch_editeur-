📋 PROBLÈME RÉSOLU
─────────────────────────────────────────────────────────────────────────────
Le router switch-case de claraApiService.ts déterminait correctement les routes
et endpoints n8n, mais ces informations n'étaient pas visibles en front-end.

L'utilisateur ne pouvait pas vérifier quel case était activé et quelle URL était
utilisée pour traiter son message.

✅ SOLUTION IMPLÉMENTÉE
─────────────────────────────────────────────────────────────────────────────
Ajout de notifications front-end qui affichent :
  1. Le numéro du case détecté (Case 1, Case 2, etc.)
  2. La route validée (routeKey : design, cia_cours, etc.)
  3. L'URL complète de l'endpoint n8n utilisé

🔧 MODIFICATIONS TECHNIQUES
─────────────────────────────────────────────────────────────────────────────

1. NOUVELLE MÉTHODE : getN8nEndpointWithInfo()
   ────────────────────────────────────────────
   Remplace getN8nEndpoint() avec retour enrichi :
   
   Interface de retour :
   {
     endpoint: string,      // URL de l'endpoint n8n
     caseName: string,      // "Case 1", "Case 2", etc.
     routeKey: string       // "design", "cia_cours", etc.
   }

2. MÉTHODE HELPER : resolveEndpointFromRouteKey()
   ───────────────────────────────────────────────
   Centralise la résolution des URLs à partir des routeKeys
   Évite la duplication de code

3. REFACTORING : getN8nEndpoint() (legacy)
   ────────────────────────────────────────
   Maintenue pour compatibilité ascendante
   Utilise maintenant getN8nEndpointWithInfo() en interne
   Marquée @deprecated

4. NOTIFICATION FRONT-END
   ──────────────────────
   Dans sendChatMessage() :
   
   ```typescript
   const routingInfo = this.getN8nEndpointWithInfo(message);
   const resolvedEndpoint = routingInfo.endpoint;
   
   // 🔔 NOTIFICATION FRONT-END
   addInfoNotification(
     `🔀 Router n8n - ${routingInfo.caseName}`,
     `Route: ${routingInfo.routeKey}\nURL: ${resolvedEndpoint}`,
     8000
   );
   ```
