🔧 CONFIGURATION TECHNIQUE
═══════════════════════════════════════════════════════════════════════════

Environnement:
  • Windows 10 + WSL2 Ubuntu 24.04.1
  • Docker Engine 27.5.1
  • Docker Compose v2.32.4

Architecture:
  • Réseau: claraverse-vfirebase_e-audit-network (bridge)
  • Volume: claraverse-vfirebase_backend-data
  • Persistance: /root/.clara dans le conteneur backend

Ports:
  • 8080 → Frontend (modifié pour éviter Flowise sur 3000)
  • 5000 → Backend

Images:
  • Frontend: ~580 MB (Node 20 Alpine + build)
  • Backend:  ~200 MB (Python 3.11 Slim)


🔑 COMMANDES ESSENTIELLES
═══════════════════════════════════════════════════════════════════════════

Démarrer:    docker compose up -d
Arrêter:     docker compose down
État:        docker compose ps
Logs:        docker compose logs -f
Redémarrer:  docker compose restart
Rebuild:     docker compose build


💡 POINTS IMPORTANTS
═══════════════════════════════════════════════════════════════════════════

1. Les conteneurs conservent leur état entre les redémarrages
2. Les données backend sont persistées dans un volume Docker
3. Le port frontend est 8080 (pas 3000) pour éviter Flowise
4. Pas besoin de rebuild sauf modification du code
5. Le backend met ~15 secondes à être "healthy"
6. Les imports Speech2Text/Text2Speech sont optionnels
