🔍 POURQUOI L'ERREUR APRÈS 3 MINUTES ?
═══════════════════════════════════════════════════════════════════════════════

Le timeout de 5 minutes dans claraApiService.ts est correct et uniforme.
Cependant, l'erreur après ~3 minutes peut être causée par:

1. ⏱️ TIMEOUT DU NAVIGATEUR
   ─────────────────────────────────────────────────────────────────────────
   Chrome/Edge: ~5 minutes par défaut
   Firefox: ~90 secondes par défaut
   Safari: ~60 secondes par défaut
   
   → Le navigateur peut couper la connexion avant le timeout configuré

2. ⏱️ TIMEOUT DU SERVEUR N8N
   ─────────────────────────────────────────────────────────────────────────
   n8n a ses propres timeouts configurables:
   - Timeout de workflow
   - Timeout de webhook
   - Timeout de node HTTP
   
   → Si n8n timeout avant 5 minutes, la requête échoue

3. ⏱️ TIMEOUT RÉSEAU/PROXY
   ─────────────────────────────────────────────────────────────────────────
   Exemples courants:
   - Nginx: 60 secondes par défaut
   - Apache: 300 secondes par défaut
   - Cloudflare: 100 secondes pour les requêtes HTTP
   
   → Un timeout intermédiaire peut couper la connexion

4. ⏱️ PROBLÈME CORS
   ─────────────────────────────────────────────────────────────────────────
   Les requêtes CORS font d'abord une requête OPTIONS (preflight)
   
   Si le preflight timeout ou échoue:
   - La requête principale n'est jamais envoyée
   - Erreur "Failed to fetch" ou "Network error"

🔧 SOLUTIONS RECOMMANDÉES
═══════════════════════════════════════════════════════════════════════════════

SOLUTION IMMÉDIATE: Diagnostic
─────────────────────────────────────────────────────────────────────────────
1. Tester avec curl pour identifier si le problème est côté serveur
2. Vérifier les logs n8n pour voir si le workflow s'exécute complètement
3. Vérifier la configuration CORS sur n8n

SOLUTION RECOMMANDÉE: Système de Polling
─────────────────────────────────────────────────────────────────────────────
Au lieu d'attendre la réponse pendant 5 minutes:
1. Envoyer la requête à n8n
2. n8n retourne immédiatement un ID de tâche
3. Le frontend poll régulièrement pour vérifier l'état
4. Quand la tâche est terminée, récupérer le résultat

Avantages:
✅ Pas de timeout
✅ Barre de progression possible
✅ Meilleure expérience utilisateur
✅ Plus robuste

SOLUTION AVANCÉE: Webhooks Asynchrones
─────────────────────────────────────────────────────────────────────────────
1. n8n envoie une notification quand le traitement est terminé
2. Le frontend écoute les notifications (WebSocket ou SSE)
3. Pas besoin d'attendre la réponse HTTP
